Welcome to OMail

OMail is a messaging system built for privacy. There's no email address to remember, no password to reset, and no personal data to collect. Your identity is protected by cryptography, your messages are encrypted end-to-end, and you control where your mailbox lives.

Why OMail?

No Accounts

Create an identity with just a passkey. No email address, no password, no recovery questions.

Private by Default

Messages are encrypted end-to-end. Only you and your contact can read them — not even OMail can.

Quantum-Safe

Uses advanced cryptography that resists both today's computers and tomorrow's quantum computers.

Sovereign

Start on a public node, then migrate to your own private server whenever you want. It's your choice.

How It Works

OMail replaces email with a completely different model:

  • Your identity is a cryptographic key (a long random number).
  • Instead of an email address, you share a User Privacy Address — a code unique to each contact.
  • Messages travel through Tor, encrypted so nobody can read them in transit.
  • The server can't see your messages, your keys, or who's talking to whom.

Getting Started

Installation

OMail runs on macOS, Linux, and Windows. You need Python 3.11 or later and Tor.

Install OMail

pip install -e .

Start the Portal

For development (without Tor):

omail --no-tor --host-name "My Node"

With Tor (requires running Tor daemon):

omail --host-name "My Sovereign Node"

Open in Browser

For --no-tor, visit http://127.0.0.1:8000 in your browser.

With Tor, the terminal prints your .onion address. Open it in Tor Browser.

Create Your Identity

Click "Create identity with a passkey" and use your device's biometric or security key.

Your identity is now live

Your First Message

Every OMail node has a built-in Host Node contact — it's like a talking clock that echoes your messages back. Try it:

  1. Open the mailbox and select "Host Node" in the contacts list.
  2. Type ping and hit Send.
  3. You'll get pong back instantly — encrypted end-to-end.

User Privacy Addresses

Email addresses are memorable but terrible for privacy — they're tied to your identity, can be harvested for spam, and expose your server to traffic analysis. OMail uses User Privacy Addresses (UPAs) instead.

What's a UPA?

A UPA looks like this:

3g4yxk5j2z8w9p1a4m7n2x9l5b6v3q8k.onion/7f2h9j4k6l1m3n8p2q5r7s9t2u4v6w8x

It's a pair of cryptographic codes separated by a slash:

  • Before the slash: Your mailbox location (which server hosts it).
  • After the slash: Your unique identity on that server.

Why They Matter

Each contact gets a different address

If you share your UPA with Alice and Bob, they each see a different code after the slash. Nobody can tell they're talking to the same person.

Impossible to enumerate

Spammers can't scrape a list of valid addresses from a server. Every UPA is completely random.

No typos ruin delivery

Each UPA includes a checksum. If you mistype even one character, the system rejects it before it goes anywhere.

Sharing Your UPA

You can share your UPA (or a QR code of it) with a contact in three ways:

  1. Copy-paste the text code (long but foolproof).
  2. Scan a QR code with your phone (fast, but out-of-band).
  3. Share an HTTPS link if you publish your UPA on your website.

Once shared, a UPA is tied to that contact. Keep different UPAs for different people.

Managing Contacts

Add a Contact

Click the + Add contact button in the sidebar and paste their UPA. Give them a name (for your records only).

That's it — no verification, no servers checking that they exist. If the UPA is valid, OMail accepts it.

Start Messaging

Click their name to open the thread. Type your message and hit Send. The message is encrypted before it leaves your device.

Delete a Contact

Click the × next to their name. Their message history on your device is deleted. They can still message you (but you won't see it unless you re-add them).

Message History

Messages are stored on your device in your encrypted vault. When you receive a message, you can archive it (move it to permanent encrypted storage) or delete it (gone forever).

Your message history never leaves your device unless you export it

Sending and Receiving Messages

Encryption Happens Automatically

Every message you send is encrypted with cryptographic keys that only you and the recipient have. OMail handles this in the background — you just type and send.

Message Routing

When you send a message to a contact on another OMail node:

  1. Your message is encrypted.
  2. It travels through Tor to their node (nobody sees the sender or receiver).
  3. Their node stores it encrypted. Their node can't read it.
  4. When they come online, they download and decrypt it.

Typing Indicators & Delivery Status

OMail does not support typing indicators or read receipts. Why? Because they leak metadata — someone watching the server can tell you're composing a message even if the message is encrypted.

Messages either arrive (and you decrypt them) or they don't. You'll see the message in your thread when the recipient receives and decrypts it (if they choose to send confirmation).

Offline Messages

If your recipient is offline, the message waits on the server encrypted. When they come online and fetch it, they decrypt and archive it. Old messages are periodically cleaned up.

Become Your Own Host

Why Self-Host?

By default, your mailbox lives on a shared public node. This is convenient, but:

  • The node operator could shut down or disappear.
  • You're dependent on their infrastructure.
  • You lose full control over your identity.

OMail lets you run your own private node — instantly, without downtime.

How It Works

Click ⚑ Become Your Own Host in the mailbox sidebar. OMail will:

  1. Generate a new independent server identity (cryptographic key).
  2. Start a Tor hidden service pointing to your device.
  3. Rewrite your UPA routing — messages now point to your server, not the public node.
  4. Print the new routing table on your terminal.

Before You Migrate

Your device must stay running to receive messages. If you turn off the server, new messages can't be delivered. Consider using a always-on computer (like a Raspberry Pi or cloud VM) instead of a laptop.

After Migration

You'll need to share your new UPA with contacts. The old UPA still works if the public node is still hosting it (you can ask them to forward), but the authoritative address is the new one.

Your mailbox is now 100% under your control. No operator can shut it down.

Frequently Asked Questions

Can OMail read my messages?

No. Messages are encrypted before they leave your device. OMail (the software and anyone running it) can never see the content.

What if I lose my passkey?

Your passkey is your identity. If you lose it (phone broken, security key lost), you can't recover your messages or identity. Back up your passkey securely (your device usually offers this via iCloud Keychain or Google Password Manager).

How do I verify someone's identity?

OMail doesn't verify that a UPA belongs to Alice or Bob. You have to verify out-of-band (call them, video chat, meet in person) and confirm they gave you the right UPA.

Can someone impersonate me?

No. Your messages are cryptographically signed with your private key. Recipients can prove you sent them.

Is my UPA visible to other users on my node?

No. Each UPA is contact-specific. Even if someone else is on the same node, they can't see your address or your messages.

Can I use OMail on multiple devices?

Your identity lives in your passkey. If your passkey is synced across devices (iCloud, Google), you can use OMail on each device. Each device maintains its own message archive.

What if I get spammed?

Block the contact. Since each contact has a different UPA, you can selectively allow/deny without affecting others. You can also migrate to your own node if you want total control.

OMail is a decentralized messaging protocol implemented as a Python/JavaScript stack. This guide covers architecture, cryptographic foundations, development setup, testing, and contributing.

Python 3.11+

Architecture

Layers

Cryptographic Layer (omail/crypto/)

  • triple_ratchet.py: DH ratchet (X25519), KEM ratchet (ML-KEM-768), symmetric chains (HMAC-SHA256 → AES-256-GCM).
  • kem.py: KEM abstraction with ML_KEM_768 class and X25519_KEM fallback.
  • convert.py: Ed25519 → X25519 birational mapping for prekey derivation.

Addressing Layer (omail/upa.py)

  • Encodes Ed25519 pubkey + SHA3-256 checksum as base32 (same format as Tor v3 onion).
  • Contact-specific UPAs derived deterministically from node key + contact identity.
  • Checksum validation prevents forged or typo'd addresses from routing.

Persistence Layer (omail/db.py)

  • SQLite schema: users, credentials, vaults, messages, user_prekeys, host_sessions.
  • Vaults and message envelopes are opaque AES-GCM ciphertext. Zero plaintext key material server-side.
  • Prekey bundles contain only public keys; private halves live in encrypted vault.

Authentication Layer (omail/webauthn.py)

  • FIDO2 registration/authentication ceremonies via python-fido2.
  • WebAuthn PRF extension: vault key is derived inside authenticator, never leaves device.
  • Fallback to device-local key (with UI warning) when authenticator lacks PRF.

Server Layer (omail/server.py)

  • aiohttp portal on 127.0.0.1:8000, accessed via Tor Hidden Service.
  • Routes: passkey ceremonies, vault ops, contact mgmt, message send/archive, WebSocket push.
  • Host-to-host ratchet replies; local blind delivery; remote federation queued.

Host Identity Layer (omail/host.py)

  • Manages node's Ed25519 key (doubles as Tor onion identity and messaging identity).
  • Publishes prekey bundles; responds to handshakes; receives encrypted envelopes.
  • Deterministic Host Node responder: ping → pong, etc.

Message Flow

Alice (Tor Browser)
  ├─ registration (passkey + PRF)
  ├─ vault encrypt (PRF key via WebAuthn)
  ├─ POST /api/contacts → Bob's UPA
  ├─ GET /api/prekeys/{bob_upa} → Bob's bundle
  ├─ JS Triple Ratchet.initiate(bob_bundle)
  ├─ POST /api/messages → encrypted envelope
  ├─ server.route_envelope(bob_upa)
  └─ WebSocket /notify push to Bob

Bob (Tor Browser)
  ├─ WebSocket notified of envelope
  ├─ GET /api/messages → encrypted envelope
  ├─ JS Triple Ratchet.decrypt(envelope)
  ├─ POST /api/messages/{id}/archive → upload vault blob
  └─ DELETE /api/messages/{id} (plaintext destroyed)

Sovereignty Migration

When user clicks "Become Your Own Host":

  1. POST /api/migrate → generates new Ed25519 key.
  2. stem provisions new Tor hidden service → publishes to node.
  3. UPA rewritten: old-host.onion/key → new-host.onion/key.
  4. Node config flipped to Host Mode; terminal prints routing table.

Triple Ratchet Protocol

Overview

OMail's encryption combines Signal's Double Ratchet with a third KEM ratchet that mixes a post-quantum shared secret (ML-KEM-768) into every epoch. The result is forward secrecy and post-compromise security against both classical and quantum adversaries.

Three Ratchets

1. DH Ratchet (X25519)

Every epoch, a fresh Diffie-Hellman exchange generates new root and chain keys:

DH_new = X25519(my_ephemeral, their_ephemeral)
root_key_new = HKDF(root_key_old, DH_new, salt="dh")

2. KEM Ratchet (ML-KEM-768)

Every epoch, a fresh ML-KEM ciphertext encapsulates a shared secret and mixes it into the root:

kem_ct, kem_ss = ML_KEM_768.encaps(their_kem_pubkey)
root_key_new = HKDF(root_key_new, kem_ss, salt="kem")

3. Symmetric-Key Ratchet (HMAC-SHA256 chains)

For each message, a symmetric chain derives one-time AES-256-GCM keys:

chain_key_new = HMAC_SHA256(chain_key, 0x01)
msg_key = HMAC_SHA256(chain_key, 0x02)
ciphertext = AES_256_GCM(msg_key, plaintext, associated_data=header)

PQXDH Handshake

Sessions begin with a hybrid handshake against a signed prekey bundle:

1. X25519(my_eph, their_eph)
2. X25519(my_eph, their_signed_prekey)
3. X25519(my_identity, their_signed_prekey)
4. ML_KEM_768.encaps(their_kem_pubkey)
↓
All mixed through HKDF → root_key, chain_key

This provides deniability, forward secrecy, and post-quantum security on day one.

Out-of-Order Delivery

Messages may arrive out of order. Skipped message keys are banked:

if msg_num > next_msg_num:
  bank[msg_num] = HMAC_SHA256(chain_key, 0x02)  # derive now, decrypt later
  chain_key = HMAC_SHA256(chain_key, 0x01)  # advance

Prevents replay; old keys are rotated out.

State Serialization

Ratchet state (root key, chain keys, KEM state) serializes to JSON for storage in encrypted vault:

{
  "root_key": "base64...",
  "chain_key_send": "base64...",
  "chain_key_recv": "base64...",
  "dh_send": {"private": "base64...", "public": "base64..."},
  "dh_recv_public": "base64...",
  "kem_alg": "ML-KEM-768",
  "kem_send_keypair": {...},
  "kem_recv_pubkey": "base64...",
  "skipped_keys": {"epoch_num": {"msg_num": "base64..."}}
}

WebAuthn & PRF Extension

Why WebAuthn?

OMail has no passwords. Authentication is pure cryptography: you prove possession of a private key via FIDO2.

  • Phishing-proof: the browser verifies the Relying Party ID, not you.
  • Hardware-backed on security keys; biometric on modern devices.
  • No password recovery flow = no recovery email = zero personal data.

The PRF Extension

The WebAuthn PRF extensioninside the authenticator, never exposed to the browser or server:

registration:
  ├─ user creates credential (biometric/key tap)
  ├─ authenticator derives PRF(credential, salt) → 32-byte secret
  └─ secret stays in authenticator

authentication:
  ├─ user taps biometric/key
  ├─ authenticator evaluates PRF(credential, salt) → same 32-byte secret
  └─ returns secret to browser (only after successful auth)

Vault Encryption

The PRF secret is used to encrypt the user's private keys and ratchet state:

vault_key = PRF_output (32 bytes)
vault_ciphertext = AES_256_GCM(vault_key, {ed25519_key, x25519_keys, ratchet_state})

The vault blob is uploaded to the server as opaque ciphertext. The server stores it but can never read it — even if compromised, the vault key remains secret inside the authenticator.

Fallback (Non-PRF Authenticators)

Some authenticators don't support PRF. OMail falls back to a device-local key:

if PRF_available:
  vault_key = PRF(credential, salt)  # never leaves authenticator
else:
  vault_key = crypto.getRandomValues(32)  # generated locally, never uploaded
  warn user: "PRF unavailable; vault key stored on this device"

API Endpoints

Authentication

POST /api/webauthn/register/begin
  → { challenge, user_id, rp_id, attestation, extensions }

POST /api/webauthn/register/complete
  ← { id, response }
  → { credential_id, user_handle }

POST /api/webauthn/authenticate/begin
  → { challenge, allow_credentials }

POST /api/webauthn/authenticate/complete
  ← { id, response }
  → { user_handle, prf_output }

Vault

GET /api/vault
  → { vault_blob }  # encrypted AES-GCM ciphertext

POST /api/vault
  ← { vault_blob }
  → 200 OK

Contacts

GET /api/contacts
  → [{ upa, name }]

POST /api/contacts
  ← { upa, name }
  → 201 Created

DELETE /api/contacts/{upa}
  → 204 No Content

Prekey Bundles

GET /api/prekeys/{target_upa}
  → { bundle }  # signed public bundle for handshake

POST /api/prekeys
  ← { bundle }
  → 201 Created

Messages

GET /api/messages
  → [{ id, envelope }]  # pending encrypted envelopes

POST /api/messages
  ← { target_upa, envelope }
  → { id }

POST /api/messages/{id}/archive
  ← { vault_blob }  # archive encrypted with vault_key
  → 200 OK

DELETE /api/messages/{id}
  → 204 No Content

Migration

POST /api/migrate
  → { new_onion, new_upa, routing_table }

WebSocket

WS /socket
  ← { type: "notify", message_id }  # new envelope arrived
  ← { type: "pong" }  # keepalive

JavaScript Interop

Byte-identical wire format verified by:

pytest tests/test_js_interop.py -v

Flow:

  1. Python Bob publishes prekey bundle.
  2. JS Alice initiates and sends two messages.
  3. Python Bob decrypts both.
  4. Bob replies; Alice receives and decrypts.
  5. Both sides survive state serialization and epoch transitions.

Contributing

Before You Start

Open an issue or discussion to scope the change. OMail is early-stage; we want to make sure contributions align with the vision.

Development Workflow

Fork & branch

git clone https://github.com/your-fork/omail
git checkout -b feature/your-feature

Make changes

Write code, add tests. Follow PEP 8; run:

pytest tests/
node scripts/browser_e2e.js

Commit with clear messages

git commit -m "Add X feature / Fix Y bug"

Push and open a PR

git push origin feature/your-feature

Reference any related issues.

Code Style

  • Python: PEP 8, type hints, async/await.
  • JavaScript: ES6+, no bundler overhead (vendor.js is pre-bundled).
  • Cryptography: Always use established libraries (pyca/cryptography, kyber-py, tweetnacl). No custom crypto.
  • Testing: Every new feature needs tests (Python + E2E if it touches the portal).

Areas for Contribution

  • Federation: Host-to-host message routing (currently scaffolded as `queued-remote`).
  • UI/UX: Portal design, accessibility, mobile responsiveness.
  • Performance: Database indexing, message batching, crypto optimization.
  • Documentation: API docs, deployment guides, security audits.
  • Clients: Desktop (Electron), mobile (React Native), CLI tooling.

Security Concerns

If you discover a security vulnerability, please email security@omail.dev instead of opening a public issue.
